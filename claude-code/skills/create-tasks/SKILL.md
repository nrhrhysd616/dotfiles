# create-tasks

対象のソースファイル群を対話的に決定し、レビュー・改善と既存実装とのギャップ分析を経て、Claude Codeが自律実行可能なタスクファイル群に分解するスキル。

## 起動方法

`/create-tasks [起点パス]`

- `/create-tasks` — 引数なし。リポジトリ全体を起点に対話的に読み込むファイルを決定する
- `/create-tasks docs/` — ディレクトリを起点に、その配下を対象に絞り込む
- `/create-tasks docs/*.md` — glob を起点に絞り込む
- `/create-tasks docs/Phase2-Specification.md` — 単一ファイルを起点にする（従来の使い方）

## 引数

- `$ARGUMENTS`: 起点となるパス・ディレクトリ・glob（任意）。省略時はリポジトリ全体を起点とする。
  いずれの場合も最終的に読み込むファイルは Phase 1 の対話で確定する。

## 実行フロー

以下の6フェーズを順に実行する。各フェーズの完了をユーザーに報告してから次へ進むこと。

---

### Phase 1: 入力ソースの対話的決定

タスク化の対象として読み込むファイルを対話で確定する。**まだファイルの中身を精読しない**（このフェーズは候補の洗い出しと絞り込みに徹する）。

1. **起点の確定**:
   - `$ARGUMENTS` があればそれを起点とする。ディレクトリ・glob・単一ファイルのいずれも可。
   - `$ARGUMENTS` がなければリポジトリ全体（`git` 管理下）を起点とする。
2. **候補の洗い出し**: 起点配下から、タスク化の元資料になりそうなファイルを幅広く収集する。以下を優先的に候補に含める:
   - 仕様書・設計書（`*.md` のうち spec / design / requirement / phase 等を含むもの、`docs/`・`specs/` 配下など）
   - README・アーキテクチャ資料・ADR
   - 関連が強い既存コード・型定義・スキーマ（仕様の前提や制約を含むもの）
   - `.gitignore` 対象や `node_modules` 等の生成物・依存物は除外する
3. **候補の一括提示**: 収集した候補を一覧で提示する。各候補には以下を添える:
   - パス
   - 種別（仕様書 / 設計 / コード / その他）
   - 含める推奨度と、その一言理由
4. **取捨選択**: ユーザーに含める / 除外するをまとめて確認する。既定として推奨度の高いものを「含める」で提示し、ユーザーが調整できるようにする。
5. **確定**: 読み込むファイルの最終リストをユーザーに提示して合意を得てから、次フェーズで初めて中身を精読する。

---

### Phase 2: ソースレビュー・改善

1. Phase 1 で確定したファイル群を読み込む
2. 以下の観点でレビューを行う:
   - **実装可能性**: 曖昧な記述・未決定事項・技術的矛盾の洗い出し
   - **網羅性**: エラーハンドリング・エッジケース・境界条件の抜け漏れ
   - **一貫性**: 用語・命名規則・データフロー上の矛盾（複数ファイル間の齟齬も含む）
3. 問題点をユーザーに提示し、対話で解決する
4. 必要に応じて仕様書・資料を更新する（ユーザーの許可を得ること）

---

### Phase 3: 技術選定の確認

1. `CLAUDE.md` および既存コードから現在の技術スタックを把握する
2. 資料中で未決定の技術選定があればユーザーに質問する
3. 確定した技術スタックを明示的にまとめる

---

### Phase 4: 既存実装とのギャップ分析

仕様・資料の要求と、既存コードベースの実装状況を突き合わせ、**まだ実装されていない差分だけをタスク化対象として抽出する**。

1. **要求の分解**: 仕様・資料から実現すべき機能・要件を項目単位で洗い出す。
2. **実装状況の調査**: 各要件について、既存コードベースを調査し次のいずれかに分類する:
   - **実装済み**: 要件を満たす実装が存在し、追加作業が不要
   - **部分実装**: 実装はあるが、不足・未対応・仕様との差分がある
   - **未実装**: 対応する実装が存在しない
   分類の根拠となるファイル・関数（`file_path:line` 形式）を必ず添える。
3. **ギャップ表の提示**: 要件 × 実装状況（実装済み / 部分実装 / 未実装）と根拠を一覧でユーザーに提示する。
4. **合意**: 「実装済み」判定に誤りがないか、部分実装の残作業の範囲が妥当かをユーザーと確認する。実装済みと判断した要件はタスク化しない。

---

### Phase 5: タスク細分化

Phase 4 で「部分実装」「未実装」と確定した差分のみを対象にタスク化する（実装済みの要件はタスク化しない）。

1. 対象の差分を、依存関係を考慮した実行順序でタスクに分解する
2. **出力先の対話確認**: タスクファイルの出力先ディレクトリを、既定案を提示したうえでユーザーに確認する（下記「出力先の決定」参照）
3. 各タスクを個別のMDファイルとして出力する
4. 各タスクファイルには以下を含めること:
   - **目的**: このタスクで何を実現するか
   - **既存実装との差分**: 現状（実装済み部分）と、このタスクで追加・変更する内容の差分
   - **依存タスク**: 先に完了すべきタスク番号
   - **実装ファイル**: 作成・変更するファイルパスの一覧（既存ファイルの変更か新規作成かを明示）
   - **実装する関数/クラス**: シグネチャと簡潔な説明
   - **テストケース**: テストすべき項目の一覧
   - **モック方針**: 外部依存のモック方法
   - **完了条件**: このタスクが完了したと判断する基準

#### 出力先の決定

出力先は必ずユーザーに確認する。既定案は以下の考え方で提示する:

- **単一の仕様書が起点の場合**: 仕様書のパスからフェーズ名を抽出し、同階層の `tasks/` ディレクトリを既定案とする。

  | 仕様書パス                     | 既定案の出力先              |
  | ------------------------------ | --------------------------- |
  | `docs/Phase1-Specification.md` | `docs/phase1/tasks/`        |
  | `docs/Phase2-Specification.md` | `docs/phase2/tasks/`        |
  | `specs/feature-auth.md`        | `specs/feature-auth/tasks/` |

- **複数ファイル・ディレクトリが対象の場合**: 対象ファイル群の共通の親ディレクトリ配下の `tasks/`（例: `docs/tasks/`）を既定案とする。共通の親が絞れない場合はリポジトリ直下の `tasks/` を既定案とする。

いずれの場合も既定案はあくまで提案とし、ユーザーが別の場所を指定できるようにする。

ファイル名は `{番号2桁}-{kebab-case名}.md` とする（例: `01-project-init.md`）。

---

### Phase 6: 概要ファイル出力

タスク出力先ディレクトリに `00-overview.md` を作成し、以下を含める:

1. **実装状況サマリ**: Phase 4 のギャップ表（要件 × 実装済み / 部分実装 / 未実装）。実装済みとしてタスク化を見送った要件も記録する
2. **タスク一覧表**: 番号・タスク名・概要
3. **依存関係図**: ASCII アートで表現
4. **並行実行可能なタスクグループ**
5. **前提条件**: 技術スタック・環境要件
6. **元資料**: このタスク群の元になったファイルの一覧（Phase 1 で確定したもの）

---

## 注意事項

- Phase 1 でファイルを確定するまで中身を精読しない。候補の洗い出しと絞り込みを優先する
- 既存実装を確認し、実装済みの要件は重複タスク化しない。部分実装は差分のみをタスク化する
- 各タスクは「Claude Codeが1セッションで自律実行可能な粒度」にすること
- タスクの粒度が大きすぎる場合はサブタスクに分割する
- テストファーストを基本とし、各タスクにテストケースを必ず含める
- 既存のプロジェクト規約（CLAUDE.md）に従った型定義・命名規則を使用する
