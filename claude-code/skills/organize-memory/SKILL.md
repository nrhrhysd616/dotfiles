# Organize Memory Skill

このプロジェクトのauto memoryを精査し、恒久化する価値のある情報を特定した上で、
CLAUDE.mdへの昇格をユーザーに提案する。CLAUDE.mdへの編集は、
ユーザーの明示的な承認を得た後にのみ行う。

## Step 1: auto memoryディレクトリを特定する

Auto memoryは`~/.claude/projects/<project>/memory/`配下に存在する。
`<project>`は、カレントディレクトリのgitリポジトリルートから導出される。

1. `git rev-parse --show-toplevel`を実行し、リポジトリルートを取得する。
   失敗した場合(gitリポジトリでない場合)は、カレントディレクトリを代わりに使う。
2. `~/.claude/projects/`の中身を一覧表示し、そのパスに対応するディレクトリ名を探す
   (Claude Codeは絶対パスをスラッグ化しており、通常`/`を`-`に置き換えた形になる)。
3. 候補が複数見つかった場合、あるいはどれにも一致しない場合は、推測で進めず、
   ユーザーに`/memory`を実行してもらい、表示されたフォルダのパスを教えてもらう。
4. 特定したディレクトリに`MEMORY.md`が存在することを確認する。
   フォルダやファイルがまだ存在しない場合は、「まだ精査できるauto memoryがない」旨を
   ユーザーに伝えて処理を終了する。

## Step 2: すべて読み込む

`MEMORY.md`(索引ファイル)を全文読み込んだ後、そこから参照されている
すべてのトピックファイル(例:`debugging.md`、`patterns.md`、
`preferences.md`など)を読み込む。トピックファイルを飛ばさないこと——
最も重複が多いのは200行の索引そのものより、こうした個別ファイルであることが多い。

## Step 3: 各エントリを分類する

見つかった個別の事実・ルールそれぞれについて、以下のいずれかに分類する:

- **昇格候補** — 複数のセッション/エントリにまたがって、
  異なる言い回しで繰り返し出てくるもの。または「常に」「絶対に」
  「XではなくYを使う」のように、一時的な観察ではなく恒久的なルールとして
  書かれているもの。あるいは新しいチームメンバーが初日から
  知っておくべき内容であるもの。
- **現状維持** — 本当に一時的で近いうちに変わりそうなもの、
  あるいは特定の状況に限定されすぎていてプロジェクト全体のルールには
  ならないもの(例:「この特定のPRのこの特定のバグは〜が原因だった」)。
- **陳腐化/矛盾** — 削除済みのファイルを参照している、
  すでに置き換えられた技術選定に言及している、解決できない相対的な
  日付表現(「昨日の修正」など)を含んでいる、あるいはより新しい
  エントリと矛盾しているもの。

このステップでは分類のみを行い、昇格や削除は行わないこと。

## Step 4: 精査結果をユーザーに提示する

「昇格候補」「陳腐化/矛盾しているエントリ」の2つの短いリストと、
それ以外(現状維持)の件数を1行で示す。各昇格候補については、
CLAUDE.mdに追記する具体的な文言案(簡潔・命令形で、既存のCLAUDE.mdの
スタイルに合わせる)と、どのCLAUDE.mdファイルに属すべきか
(プロジェクト直下の`./CLAUDE.md`か、そのコードベースが
すでに`.claude/rules/`のようなパス限定ファイルの仕組みを使っているなら
そちら側か——これは決めつけずに事前に確認すること)を示す。

各候補について、承認・却下・修正をユーザーに求める。ユーザーから
明示的な承認を得るまで、項目ごとにStep 5には進まない
(ユーザーが一括で「全部OK」と言うのであればそれでよい)。

## Step 5: 変更を適用する

承認された昇格については:

1. 該当するCLAUDE.mdファイルに、既存のフォーマット・セクション構成に
   合わせてエントリを追加する。セクション見出しを重複させないこと——
   既存の最も近いセクションの下に追記し、どうしても当てはまるものが
   ない場合にのみ新しいセクションを作る。
2. 昇格によって不要になったエントリを、auto memoryのトピックファイル
   (または`MEMORY.md`に直接書かれていた場合はそちらから)から削除し、
   同じ情報が2箇所に残らないようにする。

承認された陳腐化/矛盾エントリについては、該当するauto memoryファイル内で
直接削除または修正する。どちらが最新かユーザーが示した場合は、
その最新の値を残す。

## Step 6: まとめを報告する

何が変わったかを報告する:昇格した件数(どのファイルに)、
削除した陳腐化エントリの件数、そして`MEMORY.md`の新しいおおよその
行数(まだ約200行を超えている場合は、その旨を警告する。毎セッション
自動で読み込まれるのは先頭200行/25KBまでのため)。

## 注意事項

- ユーザーの明示的な承認(項目ごと、または一括)なしにCLAUDE.mdを
  編集しないこと。
- CLAUDE.mdの行数は少なく高密度であることを優先する——auto memoryの
  生の文言をそのままコピーするのではなく、候補エントリを簡潔に
  書き直すこと。
- 昇格後も`MEMORY.md`が約200行を超えている場合は、
  どのエントリが実質的に将来のセッションから見えなくなっているか
  (200行/25KBを超えた部分)をユーザーに伝え、リスクを把握してもらうこと。
